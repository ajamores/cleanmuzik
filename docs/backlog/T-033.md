# T-033 — Boot reconciliation strands a review whose resolve was in flight at restart

> **Backlog (found post-R1).** Not a T-020 defect; **pre-existing on `main`** (untouched by the
> T-020 descope). Triage into a release when the fix is scheduled. See
> `docs/backlog/README.md` for how this directory works.

- **Status:** todo — filed 2026-07-23. Real bug, surfaced by an adversarial design-defect pass
  during T-020's step-back review. **HIGH:** silently strands a parked song out of the UI's reach
  (no data loss — the row and staging file are intact).
- **The finding.** The two boot sweeps in `JobWorker.start` (`server/app/jobs.py`) don't
  coordinate. `fail_unfinished_jobs()` sets every `queued`/`running` job to `error`;
  `reset_resolving_reviews()` sets every `resolving` review back to `pending`. If the process
  restarts while a resolve is **in flight** — durable state `job=running` + `review=resolving`, a
  wide window because a resolve can wait minutes behind a running download — boot produces
  `job=error` **and** `review=pending`.
- **Why that's the orphan.** This is exactly the job/review disagreement **T-029** fixed on the
  live re-park door (`_repark_after_release` settles the job to `review`, not `error`, so the two
  agree), recreated through the **boot door** T-029 didn't touch. The only UI surface for a review
  is its job's track card; there is **no standalone queue view**. A card reconnecting via
  `GET /api/jobs/{id}` sees `status=error`, renders a dead error, and closes — the `status=="review"`
  review-id recovery in `routes/jobs.py` never fires. So the still-`pending` review is unreachable
  from the product, even though `GET /api/reviews` would list it and the staging file is intact.
- **Reachability.** The owner's `--reload` dev server restarting mid-resolve hits this; so does any
  crash / redeploy on the always-on Phase-1 host.
- **Fix shape (don't pre-commit).** Mirror `_repark_after_release` in boot reconciliation: reconcile
  reviews first, then fail only jobs with **no** pending review — or re-`review` any job whose reset
  review points back at it. Add a boot-path test driving the two sweeps together against a
  `running`-job + `resolving`-review pair; none exists today (`test_worker_start_fails_orphaned_jobs`
  uses a bare running job with no review).
- **Done when:** a restart with an in-flight resolve leaves `job` and `review` in **agreement**
  (the job settles to `review`, the card recovers the panel), covered by a test of the two-sweep
  interaction.
