# T-032 — Browser reload loses all job cards (no restore-on-load)

> **Backlog (post-R1).** Not R1 scope; does not gate R1 close. Triage into a future
> release when the R2/UI story is scoped (see `docs/roadmap.md`). See
> `docs/backlog/README.md` for how this directory works.

- **Status:** todo — **latent, owner-deferred (2026-07-21).** Surfaced during T-020 browser
  `/verify`. Not a T-020 defect and not an R1 blocker — R1 never promised reload survival.
- **The finding.** `client/src/App.tsx` holds the job list in React component state only. A
  browser reload drops every card; the jobs still exist server-side (their durable `jobs` row),
  but the UI has no path to re-hydrate them, so they vanish from view.
- **Why it's real but not urgent.** Single-user tool, and the durable job state (the `jobs` row)
  survives — nothing that matters is *lost*, only unshown. (Note: since ADR-015 the landing
  path/tags are NOT durable — they ride the terminal SSE event — so a re-hydrate would show a
  bare status for a landed job, which is the same accepted best-effort-path cost.) It becomes
  worth fixing once the UI is expected to be long-lived across reloads/restarts.
- **Directions to weigh (don't pre-commit):** (a) a `GET /api/jobs` list endpoint the client
  calls on mount to re-hydrate; (b) client-side persistence (localStorage) as a lighter stopgap;
  (c) fold into the broader R2 UI pass rather than a standalone fix.
- **Done when:** a decision is recorded here, and if built, a browser reload re-shows in-flight
  and recently-landed jobs — verified in a browser.
